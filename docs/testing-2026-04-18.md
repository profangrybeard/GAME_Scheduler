## PR #25 — "Building xlsx…" during Export solve phase (commit `b891a74`)

What landed: when the user clicks Export, the SolveProgress title now
reads `"Building xlsx…"` from event #1 instead of `"Solving…"`. Fixes
the "did I misclick Generate?" confusion.

### Reproduction of the original bug

1. Open the app, set up a valid quarter, run Generate once (so caches
   exist).
2. Click **Export**.
3. Watch the SolveProgress panel title during the solve phase (first
   ~5–30s).

**Old behavior:** title said `"Solving…"` — indistinguishable from
Generate. User briefly thinks they hit the wrong button.

**New expected behavior:**

- [ ] Title reads **`"Building xlsx…"`** from event #1 through the end
      of the solve phase
- [ ] When backend emits `xlsx_writing`, title flips to
      `"Writing Excel…"`
- [ ] On `export_complete`, title reads `"Export complete"`
- [ ] XLSX file downloads as normal

### Regression checks

- [ ] Clicking **Generate** (not Export) still shows `"Solving…"`
      throughout the solve phase. Nothing in the Generate path changed.
- [ ] After Export completes, clicking a different mode card in
      SolveProgress still flips the calendar (PR #24 behavior unchanged).

## PR #26 — DnD on solver-assigned cards actually works (commit `03ccfe8`)

What landed: `pinToSlot` now clears `assignment` alongside `pinned`.
Fixes the reported calendar → roster drag bug **and** two latent bugs
(cell→cell and cell→dock drags on solver-placed cards).

### Reproduction of the original bug

1. Run Generate and wait for the solve to complete. The grid shows
   all modes' assignments.
2. Drag any placed class card **from the calendar** toward the
   **roster panel** (left side).
3. Release on the roster list.

**Old behavior:** the roster highlighted blue on dragover (drop target
recognized), but on release nothing happened — the card stayed on the
calendar. Drop appeared to fail silently.

**Why it was broken:** `pinToSlot(id, null)` only set `pinned: null`.
The calendar renders via `effectiveSlot = assignment ?? pinned`, so a
stale `assignment` kept the card on the grid regardless.

**New expected behavior:**

- [ ] Drop fires, card **appears in the roster list**, **disappears
      from the calendar grid**.

### Extended coverage (same root cause, other affected flows)

- [ ] **Calendar cell → Roster:** card unplaces ✅ (above)
- [ ] **Calendar cell → Dock** (the drop zone just below the Quarter
      Schedule grid, if visible in your layout): card unplaces.
- [ ] **Calendar cell → Another calendar cell:** card moves to the new
      slot. Previously this was also a silent no-op for solver-placed
      cards.

### Regression checks

- [ ] **Manually pinned cards still work:** before running Generate,
      drag a card from the Catalogue/roster onto a cell, then drag it
      to another cell. Card moves as expected.
- [ ] **Pin → unpin round trip:** manually pin a card, then drag it
      back to the roster. Card returns to the unplaced list.

### Known behavior (not a bug, not new)

Clicking a different mode card in SolveProgress **after** manually
moving cards will re-apply that mode's cached assignments and wipe
your manual placements. This is pre-existing — `applyModeAssignments`
always replaces assignments wholesale on mode switch. Not introduced
by any of today's PRs.

## If anything fails

1. Check the browser console for React errors.
2. Check the Fly logs: `fly logs -a game-scheduler` (or whatever the
   app name is — see `DEPLOY.md`).
3. The fresh deploy from the #26 merge may need a minute to propagate.
   Hard-refresh (Cmd/Ctrl+Shift+R) clears stale JS bundles.
4. localStorage can mask bugs in preview mode — if behavior looks
   impossible, clear site data and reload (see
   `reference_preview_state_debug.md` in memory).
