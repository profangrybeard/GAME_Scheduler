## PR #33 — Gear UI prototype (commit `a78c356`)

What landed: a radial-gear mixer on the Tune card inside SolveProgress.
Three dials (Affinity / Time / Overload) that sum to 100, with
`MIN_FLOOR = 5` so no dial can be zeroed. Values persist to localStorage
(key `tunedWeights`). The gear **only** renders on the Tune card, and the
Tune card only exists after the first successful Generate — this is
intentional progressive disclosure, not a discoverability bug.

## PR #35 / PR #37 — Generate button was passing MouseEvent into `requestSolve` (commit `32adb33`)

What landed: `onSolve={() => { void requestSolve() }}` at the boundary in
`App.tsx`, so the click handler cannot smuggle a MouseEvent into the
optional `overrideMix` arg.

### Reproduction of the original bug

1. On a build with the bug, click Generate.
2. Network tab → `/api/solve-stream` returns **422 Unprocessable Entity**.
3. Error body: `{"detail":[{"type":"missing","loc":["body","tunedWeights","affinity"],...}]}`

### Why it was broken

`requestSolve(overrideMix?: Mix, displayMode?: SolveMode)` has wider
arity than the `onSolve: () => void` prop it was satisfying. TypeScript
permits a wider function to satisfy a narrower function type (trailing
args are ignored at the type level) — but at runtime React invokes the
click handler with a MouseEvent, and that event landed in the
`overrideMix` slot:

- `overrideMix ?? tunedMix` → event object is truthy, so `tunedMix` is
  never consulted.
- `mixToSolverWeights(event)` read `.affinity`, `.time`, `.overload` off
  the event — all three were `undefined`.
- `JSON.stringify` silently drops undefined values.
- Request body: `tunedWeights: {}`.
- Pydantic's `TunedWeightsModel` requires all three fields → 422.

### Expected new behavior

- [ ] Generate returns 200 and streams SSE events normally.
- [ ] Network inspector on the `/api/solve-stream` POST request: body
      contains `tunedWeights: {"affinity": <int>, "time_pref": <int>,
      "overload": <int>}` with numeric values matching the gear state.

## PR #36 — Normalize tuned weights to canonical scale (commit `f22fc5f`)

What landed: `_normalize_weights()` in `solver/objectives.py` rescales
any incoming weight vector so its max equals 10 (the canonical
`MODE_WEIGHTS` ceiling), preserving the user's intended ratio.
`{5, 90, 5}` → `{1, 10, 1}`. No-op for the canonical presets.

### Reproduction of the original bug

1. On a build without the fix (e.g., SHA `63f4879`), set the Tune mix to
   `{affinity: 5, time: 90, overload: 5}` via the gear dials.
2. Click Generate.

**Old behavior** (observed in production on 2026-04-18 evening):

- Affinity: FEASIBLE 20/20, 13 tries ✅
- Time Pref: FEASIBLE 20/20, 7 tries ✅
- **Tune: UNKNOWN 0/20, 0 tries** ❌ — solver hit its 10s time limit
  without finding a single feasible assignment.
- Total solve wall time: 38s (≈ 3 × 10s timeout + model-build overhead).

### Why it was broken

The gear UI sends weights on a percent-of-100 scale (dials sum to 100,
`MIN_FLOOR = 5`). An extreme mix like `{5, 90, 5}` produces per-assignment
objective coefficients of `5 × aff_pen + 90 × time_pen` ≈ **465** — about
**9× the canonical ceiling** (~50 for the built-in `MODE_WEIGHTS`, whose
weights max at 10).

CP-SAT's search heuristic is sensitive to objective magnitude. On Fly's
throttled `shared-cpu-1x`, the inflated coefs dominated branching so
completely that the solver never backtracked to **one** feasible
assignment within its 10s budget. Locally the bug was invisible — a
multi-core desktop CPU chewed through the search fast enough anyway.

The two preset modes were unaffected because they use small canonical
weights and never pass through `tuned_weights`.

### Expected new behavior

- [ ] With Tune mix `{5, 90, 5}`, **all three** modes solve FEASIBLE 20/20.
- [ ] Tune's Penalty value is comparable to Time-Pref's (both optimize
      heavily for time) — should land in the ~20–30 range on defaults,
      not `—`.
- [ ] Tries > 0 on the Tune card.
- [ ] Slide the dials through extremes in both directions — Tune never
      times out with UNKNOWN.

## Known residuals (not blockers)

- **Fly `shared-cpu-1x` is still the floor.** The 10s per-mode budget
  (`_SOLVER_TIME_LIMIT` in `solver/scheduler.py`) is comfortable at the
  current 16-offering scale, but if a quarter grows past ~25 sections or
  the model gets new hard constraints, we may need to bump the limit or
  upgrade the VM.
- **`react-refresh/only-export-components` gotcha.** That lint rule
  forced the split of non-component exports out of `SolverTuning.tsx`
  into a sibling `SolverMix.ts`. If similar lint errors crop up on a
  component file, it's because the file exports something other than
  React components — move those exports to a `.ts` sibling.
- **Nested `<button>` restriction.** The gear button lives inside the
  Tune mode card, which used to be rendered as a `<button>`. HTML forbids
  nested interactive elements — when `isTuneCard`, the card now renders
  as `<div role="button" tabIndex={0}>` with keyboard handlers. Keep
  this in mind if refactoring SolveProgress.

## If anything fails

1. Check the browser console for React errors.
2. Network tab → `/api/solve-stream` should return 200 and stream SSE
   events. A 422 means a request-body shape regression (see PR #35/#37).
3. If the Tune card shows UNKNOWN + 0 tries but the other two modes
   solve fine, suspect the weight normalization has regressed (see
   PR #36).
4. Fly logs: `fly logs -a scad-class-scheduler` (see `DEPLOY.md`).
5. localStorage can mask bugs — clear site data and hard-refresh
   (Cmd/Ctrl+Shift+R) if behavior seems impossible.
